# Branch `20260404-w-1` вЂ” Change Summary

## Bug Fix: Template not loading on first planning workpaper open

**Problem:** When opening a planning workpaper for the first time (lazy-create), the template didn't load. It only loaded on the second open.

**Root cause:** Race condition вЂ” `getOrCreatePwp.mutate()` invalidated the query and immediately opened the overlay before the query refetched, so the overlay read stale data where `content` wasn't properly set to `null`.

| File                                      | Change                                                                                                                                                                  |
| ----------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `src/app/(main)/engagement/[id]/page.tsx` | Changed `handleOpenPlanningWp` to `async` and replaced `.mutate()` with `await .mutateAsync()` in the lazy-create path to wait for query refetch before opening overlay |

## Bug Fix: Planning workpapers not using correct workflow binding (missing subType)

**Problem:** Even after fixing cache invalidation, the "PhбєЎm vi vГ  Mб»Ґc tiГЄu" workpaper view mode still showed the old default 2-step workflow instead of the new bound 3-step workflow. The DB binding was correct, but the UI wasn't using it.

**Root cause:** `InlineWorkpaperViewer` in `PlanningTab.tsx` (view mode) was not passing the `subType` prop, so it defaulted to `""`. The backend's `resolveWorkflowId()` then fell back to the base `planning_workpaper:""` binding (or default workflow) instead of the per-step binding like `planning_workpaper:scope`.

| File                                                      | Change                                                                                                                                                      |
| --------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `src/features/engagement/components/tabs/PlanningTab.tsx` | Added `subType={step.key}` prop to all 3 `InlineWorkpaperViewer` instances: scope (line 262), understanding (line 505), and custom/generic steps (line 691) |

## Bug Fix: Workflow chart dialog showing wrong workflow

**Problem:** Even after fixing the signoff bar to show correct workflow, the workflow chart dialog still showed the old default 2-step workflow instead of the bound 3-step workflow.

**Root cause:** `WorkflowChartDialog` component only looked up workflows by `entityType`, not considering `subType`. It was missing the same lookup logic as the backend's `resolveWorkflowId()`.

| File                                                                   | Change                                                                                                                                           |
| ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `src/components/shared/workpaper/WorkflowChartDialog.tsx`              | Added `subType` prop and implemented 3-tier lookup: exact match (`entityType:subType`) в†’ base type fallback (`entityType:""`) в†’ default workflow |
| `src/features/engagement/components/tabs/PlanningWorkpaperOverlay.tsx` | Pass `subType={stepConfigKey ?? ""}` to `WorkflowChartDialog`                                                                                    |
| `src/features/engagement/components/tabs/InlineWorkpaperViewer.tsx`    | Pass `subType={subType}` to `WorkflowChartDialog`                                                                                                |
